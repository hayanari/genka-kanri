# 備品申請機能 実装ガイド（Phase 1 → 3）

## 概要

全社員が利用する「備品申請」機能を追加する。
案件（Project）とは独立したトップレベル機能として実装する。

- **Phase 1**: 備品申請の画面・フォーム・一覧（権限チェックなし、信頼ベース運用）
- **Phase 2**: ユーザーに役職(role)を追加し、承認ステップを該当役職のみ操作可能にする
- **Phase 3**: 備品申請と案件の紐づけをオプション化する

---

## 現在のアーキテクチャ（理解必須）

### データ保存方式
- Supabase の `genka_kanri_data` テーブル、`id='default'` の1行に **全データを JSONB で保存**
- `app/page.tsx` の `data` state に全データを持ち、`setData()` すると 150ms デバウンスで自動保存
- データの流れ: `setData()` → `useEffect` → `saveData()` → Supabase upsert

### ナビゲーション方式
- `app/page.tsx` 内で `view` state（"dashboard", "list", "detail" 等）を切り替える SPA 方式
- サイドバーの `navItems` 配列にメニュー項目を定義
- `/schedule` のみ別ルート（別テーブル使用のため）

### 認証
- Supabase Auth（メール+パスワード）
- `AuthGuard` コンポーネントで保護
- 管理者判定: `tokito@tokito-co.jp` のメールアドレスのみ（ハードコード）
- **役職・ロールの概念は現在ない**

### Step 1: 型定義を追加（`lib/utils.ts`）

ファイル末尾に以下の3つの型を追加する:

```typescript
/** 備品申請の明細行（最大4行） */
export interface EquipmentItem {
  id: string;
  /** 区分: 備品 or 消耗品 */
  category: "備品" | "消耗品";
  /** 品種: 文具・トナー・印刷用紙・その他 */
  itemType: string;
  /** メーカー */
  maker: string;
  /** 商品名 */
  productName: string;
  /** 品番 */
  productCode: string;
  /** 購入単価 */
  unitPrice: number;
  /** 数量 */
  quantity: number;
  /** 単位: 本・個・冊・枚・台・箱・その他 */
  unit: string;
}

/** 承認ステップ */
export interface ApprovalStep {
  id: string;
  /** 役割: リーダー / 課長 / 部長（決裁） / 経理担当 / 申請者本人 */
  role: string;
  /** 承認者名 */
  approverName: string;
  /** 結果 */
  result: "pending" | "approved" | "rejected" | "confirmed" | "";
  /** コメント */
  comment: string;
  /** 承認日時 (ISO8601) */
  actedAt: string;
}

/** 備品申請 */
export interface EquipmentRequest {
  id: string;
  /** 標題 */
  title: string;
  /** 部署 */
  department: string;
  /** 申請者名 */
  applicant: string;
  /** 申請日 (YYYY-MM-DD) */
  appliedAt: string;
  /** 購入予定日 (YYYY-MM-DD) */
  purchasePlannedDate: string;
  /** 明細（最大4行） */
  items: EquipmentItem[];
  /** 利用目的 */
  purpose: string;
  /** 備考 */
  notes: string;
  /** 小計（自動計算: 各行の 単価×数量 合計） */
  subtotal: number;
  /** 消費税（自動計算: subtotal × 10%） */
  tax: number;
  /** 支払金額合計（自動計算: subtotal + tax） */
  total: number;
  /** ステータス */
  status: "draft" | "pending" | "approved" | "rejected" | "confirmed";
  /** 承認履歴（5ステップ: リーダー→課長→部長→経理→申請者） */
  approvalHistory: ApprovalStep[];
  /** 紐づく案件ID（Phase 3 で使用。Phase 1 では空文字） */
  projectId: string;
}
```

## Phase 2: ユーザー役職（role）と承認権限

### 目標
- ユーザーにロール（一般 / リーダー / 課長 / 部長 / 経理 / 管理者）を持たせる
- 承認ステップごとに該当ロールのユーザーのみが承認・否認操作を行える
- 一般ユーザーは申請の作成と自分の申請の閲覧のみ

### 変更概要

#### Supabase 側
- `user_profiles` テーブルを新規作成（またはSupabase Auth のメタデータを使用）:
  ```sql
  CREATE TABLE user_profiles (
    id UUID PRIMARY KEY REFERENCES auth.users(id),
    display_name TEXT NOT NULL,
    department TEXT,
    role TEXT NOT NULL DEFAULT 'general',  -- general / leader / manager / director / accounting / admin
    created_at TIMESTAMPTZ DEFAULT now()
  );
  ```
- RLS ポリシー: 自分のプロフィールは読み書き可、他人のは読み取りのみ
- 管理者はアカウント管理画面（`/admin`）で全ユーザーのロールを設定

#### フロントエンド側
- `lib/supabase/auth.ts` に `getUserProfile()` / `updateUserProfile()` を追加
- ログイン後にプロフィールを取得し、Context または state で保持
- `EquipmentRequestForm.tsx` の承認タブ:
  - 各ステップのボタンは該当ロールのユーザーにのみ表示
  - 承認者名はログインユーザーの `display_name` を自動入力
  - 承認日時は操作時に自動設定
- `EquipmentRequestList.tsx`:
  - 一般ユーザー: 自分が申請者の申請のみ表示（または全件閲覧可・編集は自分のみ）
  - 承認者ロール: 自分が承認すべきステップがある申請を「承認待ち」として強調表示
- `/admin` ページにユーザーロール管理UIを追加

#### ロールと承認ステップの対応

| 承認ステップ | 必要ロール |
|---|---|
| 1. リーダー | `leader` 以上 |
| 2. 課長 | `manager` 以上 |
| 3. 部長（決裁） | `director` 以上 |
| 4. 経理担当 | `accounting` |
| 5. 申請者本人（確認） | 申請者自身 |

## Phase 3: 案件との紐づけ（オプション化）

### 目標
- 備品申請の作成時に、案件を選択して紐づけることができる（任意）
- 案件詳細画面からも、その案件に紐づく備品申請を参照できる
- 案件に紐づかない全社的な備品申請もそのまま可能

### 変更概要

#### `EquipmentRequestForm.tsx`
- フォームに「案件」選択ドロップダウンを追加（任意入力）
- 選択肢: 「なし（全社共通）」+ アクティブな案件リスト
- 選択時に `projectId` を設定

#### `EquipmentRequestList.tsx`
- 一覧に「案件」列を追加
- 案件名でのフィルタを追加

#### `components/ProjectDetail.tsx`
- 「備品申請」タブを追加（zip の STEP 4-5 をこの段階で実装）
- 表示対象: `equipmentRequests.filter(r => r.projectId === project.id)`
- 一覧のみ表示（編集は備品申請ページへリンク）

#### Props の追加
- `ProjectDetail` に `equipmentRequests` と参照用リンクを渡す
- `app/page.tsx` で `ProjectDetail` の呼び出し箇所を更新

## 参考: zip ファイルとの差分

zip ファイル（`equipment_request_feature.zip`）は「案件詳細のタブとして追加する」設計だった。
本ガイドでは以下を変更している:

| 項目 | zip の設計 | 本ガイドの設計 |
|---|---|---|
| 配置場所 | ProjectDetail のタブ | 独立したトップレベルビュー |
| `projectId` | 必須（案件に紐づく） | Phase 1 では空文字、Phase 3 でオプション化 |
| 一覧画面 | ProjectDetail 内のテーブル | 専用の EquipmentRequestList コンポーネント |
| 削除機能 | なし | あり（Phase 1 で実装） |
| 権限管理 | なし | Phase 2 で実装 |
| ナビゲーション | タブ切り替え | サイドバーメニュー |

zip ファイルの `3_EquipmentRequestForm.tsx`（フォームコンポーネント）はそのまま流用可能。
型定義（`1_types_to_add_to_lib_utils.ts`）もほぼそのまま使える（`projectId` のデフォルト値のみ変更）。
